# Provenance Guard

A backend attribution service that a creative-sharing platform can plug in to
answer one question honestly: *was this text written by a human, or generated by
AI?* It classifies submitted text with **two independent detection signals**,
reports a **calibrated confidence** (not just a binary label), surfaces a
**plain-language transparency label** to readers, and gives creators a real
**appeals** path when the system gets it wrong.

The design principle throughout: **on a writing platform, wrongly branding a
human's work as AI is the worst possible outcome.** The confidence scoring, the
asymmetric label thresholds, and the appeals workflow all bend toward *not*
falsely accusing a human.

> Full design rationale and the pre-implementation spec live in
> [planning.md](planning.md). This README documents what was built and how.

---

## Quickstart

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Optional but recommended: a real Groq key activates the LLM signal.
# Without it, the system degrades gracefully to stylometry alone (see below).
echo "GROQ_API_KEY=your_real_key" > .env

# Seed the audit log with sample decisions + an appeal, and print the
# confidence/label variation check:
python seed_demo.py

# Run the API:
python run.py     # -> http://127.0.0.1:5000
```

Then open **http://127.0.0.1:5000** in a browser for the reader-facing **demo UI**:
paste text, hit *Analyze* to see the live transparency label (color-coded by
variant, with a confidence meter and signal breakdown), and *Appeal* to flip the
status to `under_review`. It's a thin page that calls the same `/submit` and
`/appeal` endpoints — the API is the product; this just shows the label in
context the way a platform would render it.

Quick smoke test:

```bash
curl -s -X POST localhost:5000/submit \
  -H 'Content-Type: application/json' \
  -d '{"text":"I could not sleep, so I made tea and watched the snow — actual snow, in April!","creator_id":"u1"}' | python -m json.tool
```

### A note on the LLM signal

The Groq LLM signal is a first-class part of the pipeline, but the system is
built to **degrade gracefully**: if no valid `GROQ_API_KEY` is set (or the call
fails), the LLM signal reports `available: false`, the pipeline falls back to the
stylometric signal alone, applies a confidence penalty (`×0.85`) to reflect the
lost redundancy, and **records that it did so** in the audit log. Every example
output in this README was produced in that offline, stylometry-only mode — so
what you see here is reproducible without any API key. Add a real key and the
full two-signal ensemble activates automatically.

## Architecture (summary)

```
POST /submit ─► rate limiter ─► [Signal 1: LLM]  ─┐
                                [Signal 2: Stylometry] ─► confidence scoring
                                                          ─► transparency label
                                                          ─► audit log ─► response

POST /appeal ─► load original ─► status = "under_review" ─► audit log ─► response
```

Full labelled diagram (both flows, with what passes along each arrow) is in
[planning.md § Architecture](planning.md#2-architecture). Code map:

| File | Responsibility |
|---|---|
| [provenance_guard/stylometry.py](provenance_guard/stylometry.py) | Signal 2 — structural statistics |
| [provenance_guard/llm_signal.py](provenance_guard/llm_signal.py) | Signal 1 — Groq LLM classifier (graceful fallback) |
| [provenance_guard/scoring.py](provenance_guard/scoring.py) | Combine signals → `ai_score` + `confidence` |
| [provenance_guard/labels.py](provenance_guard/labels.py) | Three transparency-label variants |
| [provenance_guard/pipeline.py](provenance_guard/pipeline.py) | Orchestrates signals → score → label |
| [provenance_guard/store.py](provenance_guard/store.py) | SQLite: submissions, appeals, audit log, analytics |

## Required features

### 1. Content submission endpoint

`POST /submit` accepts text and returns a structured decision including the
attribution, confidence score, and the transparency-label text.

**Request**
```json
{ "text": "…the piece to analyze…", "creator_id": "u1", "content_type": "text" }
```
**Response (`200`)** — abridged. `content_id` is a UUID; save it to file an appeal:
```json
{
  "content_id": "6da2444c-84a6-4f05-b0b0-a25170dbcf59",
  "status": "classified",
  "attribution": "likely_human",
  "confidence": 0.6538,
  "ai_score": 0.1154,
  "label": {
    "variant": "high_confidence_human",
    "title": "✍️ Likely human-written",
    "body": "Our checks found no strong signs of AI generation (confidence: 65%). …"
  },
  "signals": [ {"name":"llm", …}, {"name":"stylometry", …} ],
  "weights": {"llm": 0.0, "stylometry": 1.0},
  "single_signal": true
}
```

### 2. Multi-signal detection pipeline (2 distinct signals)

Two signals that capture **genuinely different properties** — one *semantic*, one
*structural* — so they fail differently and their agreement is itself
informative.

**Signal 1 — LLM classifier (Groq `llama-3.3-70b-versatile`).**
Asks the model to holistically judge how AI-like the text reads (cliché density,
generic "assistant" phrasing, tonal uniformity, whether the voice feels
idiosyncratically human) and return `ai_probability ∈ [0,1]` + a one-sentence
rationale. *Captures* semantic/stylistic coherence a statistic can't see.
*Blind spot:* no ground truth (reflects the model's bias), can flag careful or
non-native human writing as AI, and is unavailable without an API key.

**Signal 2 — Stylometry (pure Python).**
Structural statistics, meaning-independent. It is itself a **weighted composite**
of four sub-metrics (this doubles as the ensemble stretch goal):

| Sub-metric | What it measures | Human ↔ AI | Weight |
|---|---|---|---|
| Sentence-length **burstiness** (coeff. of variation) | how much sentence length varies | humans bursty, AI uniform | 0.40 |
| **Lexical diversity** (root type-token ratio) | vocabulary range, length-normalized | humans wider | 0.20 |
| **Punctuation variety** (distinct marks used) | dashes/semicolons/parentheticals | humans mix more | 0.20 |
| **Human-voice markers** | contractions, first person, hedges | human tells | 0.20 |

Each yields a `human_likeness ∈ [0,1]`; the stylometric
`ai_probability = 1 − weighted_mean(human_likeness)`.

**Combination:** `ai_score = 0.6·p_llm + 0.4·p_stylo`. If the LLM is unavailable,
stylometry takes full weight and a single-signal confidence penalty applies.

### 3. Confidence scoring with uncertainty

We separate **direction** from **certainty**:

- `ai_score` = P(AI) — *which way* the evidence leans.
- `confidence` — *how much to trust* that lean.

```
distance    = |ai_score − 0.5| · 2              # how far from a coin flip
agreement   = 1 − |p_llm − p_stylo|             # do the signals concur?
reliability = min(1, words / 40)                # enough text to judge?
confidence  = distance · (0.6 + 0.4·agreement) · reliability · single_signal_penalty
```

**What 0.5 means to a user:** "we have a lean, but treat it as a hint." A `0.51`
result and a `0.95` result produce *different labels* — `0.51` falls to
**uncertain**, `0.95` produces a high-confidence label. There is no binary flip
at 0.5.

**How we tested that scores are meaningful** — `python seed_demo.py` runs four
deliberately different inputs and prints their scores. Real output (offline,
stylometry-only mode):

| Input | attribution | ai_score | confidence | label |
|---|---|---|---|---|
| Clearly AI (uniform, generic) | likely_ai | 0.85 | **0.60** | uncertain |
| Clearly human (bursty, personal) | likely_human | 0.12 | **0.65** | high_confidence_human |
| Ambiguous (edited, mixed) | uncertain | 0.61 | **0.18** | uncertain |
| Short repetitive poem | likely_ai | 0.80 | **0.15** | uncertain |

**Two example submissions with noticeably different confidence** (actual scores
from the run above):

- **Higher-confidence case (confidence 0.65 → `high_confidence_human`):**
  > *"I didn't sleep. The radiator clanked all night like somebody trapped behind
  > the wall, and by four I'd given up. So I made tea … Snow. Actual snow, in April!
  > I laughed out loud, alone, at nothing. … I don't know. Maybe I was just tired."*

  Bursty sentence lengths, contractions, hedges, varied punctuation → `ai_score`
  0.12, confidence **0.65**, labeled **✍️ Likely human-written**.

- **Lower-confidence case (confidence 0.18 → `uncertain`):**
  > *"The project began with a simple idea, but it grew complicated fast. We wanted
  > to help people, and honestly we underestimated the work. There were setbacks. …
  > iterating on the design and gathering feedback from early users along the way."*

  Middling burstiness and diversity, few voice markers → `ai_score` 0.61 (a weak
  AI lean) but confidence collapses to **0.18**, so it's labeled **❓ Attribution
  uncertain** rather than either extreme.

Distinct confidence values `[0.15, 0.18, 0.60, 0.65]` — not a binary flip.
Two things worth noticing, both intentional:
- The "clearly AI" text leans AI strongly (0.85) but only reaches **0.60**
  confidence *because a second signal wasn't available* — so it stays
  **uncertain** rather than being branded AI. That is the false-positive
  aversion working: we won't accuse without corroboration.
- The short poem and the ambiguous text collapse to low confidence via the
  `reliability` and `agreement` terms respectively.

### 4. Transparency label — three variants (verbatim)

The label is written for a **non-technical reader**. It states the result in
plain language, always says confidence is an *estimate not a certainty*, and
(for the AI case) points to the appeal path. The thresholds are **asymmetric**:
calling something AI needs confidence ≥ **0.75**; calling it human needs ≥
**0.65** — because a false "AI" accusation is the costlier error.

These are the exact strings the system emits (`{pct}` = `round(confidence·100)`),
verified reachable by [provenance_guard/labels.py](provenance_guard/labels.py):

| Variant | Shown when | Title | Body |
|---|---|---|---|
| **high-confidence AI** | `ai_score ≥ 0.65` **and** `confidence ≥ 0.75` | `🤖 Likely AI-generated` | `Our checks suggest this text was probably generated by AI (confidence: {pct}%). This is an automated estimate, not a certainty. If you wrote this yourself, you can appeal and a human will review it.` |
| **high-confidence human** | `ai_score ≤ 0.35` **and** `confidence ≥ 0.65` | `✍️ Likely human-written` | `Our checks found no strong signs of AI generation (confidence: {pct}%). This is an automated estimate and not a guarantee of authorship.` |
| **uncertain** | anything else | `❓ Attribution uncertain` | `Our checks were inconclusive for this text (confidence: {pct}%). We can't reliably say whether it was written by a human or AI, so we're not labeling it either way. Short pieces and unusual styles are especially hard to judge.` |

Rendered examples (from the label builder, at 88% / 82% / 34% confidence):

> **🤖 Likely AI-generated**
> Our checks suggest this text was probably generated by AI (confidence: 88%). This is an automated estimate, not a certainty. If you wrote this yourself, you can appeal and a human will review it.

> **✍️ Likely human-written**
> Our checks found no strong signs of AI generation (confidence: 82%). This is an automated estimate and not a guarantee of authorship.

> **❓ Attribution uncertain**
> Our checks were inconclusive for this text (confidence: 34%). We can't reliably say whether it was written by a human or AI, so we're not labeling it either way. Short pieces and unusual styles are especially hard to judge.

### 5. Appeals workflow

`POST /appeal` lets a creator contest a classification.

- **Who:** the creator of a submission (via `creator_id`). Any classification is
  appealable — most importantly a `likely_ai` call on genuine human work.
- **Input:** `{ "content_id": "...", "creator_id": "...", "creator_reasoning": "..." }`
  (`creator_reasoning` is required).
- **On receipt the system:** (1) verifies the content exists (`404` if not);
  (2) sets the submission's `status` to **`under_review`**; (3) writes an
  `appeal` audit-log entry that includes the creator's reasoning **and a snapshot
  of the original decision** so the record is self-contained; (4) returns a
  confirmation + the appeal record. Re-classification is intentionally **manual**.
- **Human reviewer view:** `GET /appeals` returns the queue — each item pairs the
  creator's reasoning with the original decision + full signal breakdown, so a
  moderator has everything needed to uphold or overturn.

```bash
curl -s -X POST localhost:5000/appeal -H 'Content-Type: application/json' \
  -d '{"content_id":"<id-from-submit>","creator_reasoning":"I wrote this myself; I am a non-native English speaker and my style is more formal than typical."}'
```

Verified end-to-end: after an appeal, `GET /submission/<content_id>` reports
`"status": "under_review"` and `GET /appeals` shows the queued item.

### 6. Rate limiting

Implemented with Flask-Limiter (per client IP). Configured in
[provenance_guard/config.py](provenance_guard/config.py); returns `429` with a
JSON body when exceeded (verified in testing).

| Endpoint | Limit | Reasoning |
|---|---|---|
| `POST /submit` | **10/min, 100/day** | A real creator posts a handful of pieces a day; 10/min tolerates the natural burst of editing and resubmitting a draft, while 100/day caps a single account far above any honest use. An adversary trying to flood the classifier (it invokes a paid LLM per call) is stopped by both the burst and daily ceilings. |
| `POST /appeal` | **5/min, 20/day** | Appeals are rare and deliberate — a genuine creator files at most a few. Tighter limits here blunt spam/harassment of the human review queue, which is the scarce resource. |
| all endpoints (default) | **200/day** | Backstop on read endpoints (`/log`, `/analytics`) to prevent scraping. |

**Evidence** — sending 12 rapid `/submit` requests (the first 10 succeed, then
the limiter trips):


The `429` body: `{"error": "Rate limit exceeded.", "detail": "10 per 1 minute"}`.

### 7. Audit log

Every decision **and** every appeal writes exactly one structured row to the
`audit_log` table (`store.py`). Entries are flat and lead with
`content_id` / `creator_id` / `timestamp` / `event_type`, then carry the decision
fields (including **both individual signal scores**) or the appeal fields. The
full nested `signals` breakdown is included too. Read it via `GET /log?limit=N`.

Sample entries (real output from `python seed_demo.py`, `signals` abridged):

## Stretch features implemented

- **Ensemble detection** — the stylometric signal is a documented weighted vote
  of four sub-metrics (table above); combined with the LLM that's a 5-input
  ensemble with explicit weights, all surfaced in `signals` and the audit log.
- **Analytics dashboard** — `GET /analytics` reports the attribution
  distribution, the **appeal rate** (appeals ÷ submissions), the label-variant
  distribution, and **average confidence** as the extra metric. Example:
  ```json

   "label_variant_distribution": {"uncertain": 3, "high_confidence_human": 1},
   "average_confidence": 0.3972}
  ```

## Anticipated edge cases (where the system is weak)

1. **Minimalist / repetitive poetry** ("so much depends / upon / a red wheel
   barrow") is statistically uniform — the fingerprint stylometry reads as AI.
   Mitigation: short + low-reliability text drags confidence down and the AI
   threshold is high, so it lands on **uncertain** rather than a false accusation
   (verified: the poem sample scores ai_score 0.80 but confidence 0.15 →
   uncertain).
2. **Very short submissions** (haiku, tweet-length) lack enough sentences for
   stable statistics; `reliability = words/40` caps their confidence.
3. **Human-edited AI text** breaks structural uniformity while keeping AI
   phrasing — the two signals *disagree*, which correctly lowers confidence via
   the agreement term. This is a case we *should* be unsure about.

## Spec reflection

**How the spec helped.** Writing the confidence formula and the three label
variants *before* any code (planning.md §3.2–3.3) forced the decision that
`confidence` and `ai_score` are two different things. Had I coded first I'd
almost certainly have conflated "how AI is it" with "how sure are we" into one
number — and the asymmetric label thresholds (AI needs ≥0.75, human ≥0.65) only
make sense once those are separate. The spec surfaced that design choice while it
was still cheap to make.

**Where the implementation diverged.** The spec's confidence formula was
`distance · agreement · reliability`. In implementation I added an
`AGREEMENT_FLOOR` term (`0.6 + 0.4·agreement` instead of raw `agreement`),
because the raw version let a single strongly-directional signal be crushed to
near-zero confidence by mild disagreement — a clearly-human piece with one
dissenting sub-signal was landing in "uncertain". The floor keeps disagreement
*meaningful* without letting it dominate. The spec is annotated to point here as
the source of truth.

## AI usage

Two concrete instances (this project was built with AI assistance per the
milestone workflow):

1. **Stylometry signal + Flask skeleton (M3).** I gave the AI the detection-signal
   section and the architecture diagram and asked for the `POST /submit` route
   stub and the first signal function. It produced a working type-token ratio but
   used *plain* TTR, which is length-biased (longer texts always look less
   diverse). I overrode it with **root TTR (Guiraud's R)** and added the
   `reliability` term for short texts, which the generated version lacked
   entirely.

2. **Confidence scoring + label mapping (M4–M5).** I asked for the combine logic
   from the uncertainty section. The generated scorer used a **symmetric** 0.5
   threshold for both labels — reasonable-looking, but it silently dropped the
   false-positive asymmetry the spec calls for. I corrected it to the split
   `HIGH_CONF_AI = 0.75` / `HIGH_CONF_HUMAN = 0.65` bars and added the
   single-signal penalty so the offline (no-LLM) path reports honestly lower
   confidence rather than pretending two signals agreed.

## API reference
